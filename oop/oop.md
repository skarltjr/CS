### 객체지향은 클래스가 아니라 변경과 메세지

### 패러다임
```
시스템을 어떻게 바라보는가?
 
절차적 패러다임 : 시스템을 함수와 데이터의 집합으로 바라본다

함수형 패러다임 : 함수의 집합

객체지향 패러다임 : 시스템을 객체의 집합으로 바라본다
```

### 객체지향
```
굉장히 작은 객체들을 모아서 시스템을 구성한다.
그리고 하나의 객체는 무조건 다른 객체와 연결된다.
왜? : 객체가 다른 객체에게 메세지를 전송하기 위해서!!

만약 메세지를 전송할 필요가 없는데 연결되어 있다면 이는 불필요한 연결

이렇게 하나의 객체가 다른 객체에서 메세지를 전달해서 상호작용하는것이 협력!!!
```

### 객체지향 정의
```
1. 시스템을 객체의 집합으로 바라보며
2. 사용자에게 기능을 제공할 때 객체들이 메세지를 주고 받아
3. 협력할 수 있도록 구현하는 것 
```

### 객체의 협력
```
런타임 측면과 컴파일타임 측면

런타임 측면 : 기능 설계 -> 객체의 협력은 시스템의 런타임 측면  
컴파일타임 측면 : 코드(클래스) 설계 -> 클래스를 바탕으로 기능을 코드로 구현

그럼 설계를 주도하는 쪽은 과연 런타임 / 컴파일 중 무엇일까?
런타임 측면 : 사용자!!에게 가치를 제공할 수 있는 기능 제공
컴파일 측면 : 개발자!!가 읽기 쉽고 변경하기 쉬운 품질 좋은 코드 제공

결국 실사용자에게 제공되는 기능이 잘 갖춰진 후 이를 잘 구현하는 방향이 중요!
즉 객체간의 협력(기능)을 잡고 이 후 코드(클래스) 설계 방향으로 가야한다.
```

### 객체 협력 설계
```
책임 - 주도 설계
중요한것은 설계를 주도하는것은 책임(매새지).

즉 책임을 결정하고 이를 수행할 객체를 선택
어떤 기능이 필요한데 이걸 누가할까의 생가 흐름

1. 책임을 결정한다.
2. 해당 책임을 수행할 객체를 선택
3. 책임에 따라 메세지를 결정
4. 스스로 처리할 수 없는 작업을 다른 객체에 요청(메세지) / 협력
5. 해당 메세지를 스스로 처리할 수 있는 객체 선택

--
1. 영화를 예매해야해!
2. 그럼 이걸 누가해야할까?
3. reserve해!
4. 이거 도움이 필요하니까 movie객체한테 calculateFee 메세지 보내자!
5. movie객체는 이거 처리할 수 있으니 처리
```

### 클래스 설계
```
메세지와 매서드
메세지 : 하나의 책임, 어떠어떠한 기능
매서드 : 메세지를 처리할 수 있는 하나의 방법

객체보다 메세지가 중요!
상태보다 행동이 중요!
```

### 추상화
```
공통 부분을 뽑아낸다.
즉 현재 문맥에서 필요사항을 뽑아내는것

클래스의 공통 기능을 뽑아내어 인터페이스로 추상화
여기서도 중요한것은 기능!. 즉 런타임 측면의 메세지, 책임
```
